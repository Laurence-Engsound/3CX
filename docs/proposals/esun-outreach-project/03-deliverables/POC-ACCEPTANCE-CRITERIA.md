## 一、POC 範圍與目的

### 1.1 目的

驗證 VOXEN + 3CX 替代方案能否在玉山實際業務場景下，達到「替換 Genesys Engage」所需的功能、效能、可用性與整合度。

## 二、驗收標準總覽

POC 驗收依「五大維度 + 25 項 criteria」進行，每項以 ✅ / ⚠️ / ❌ 三段式判定：

| 維度 | Criteria 數 | 必過項數 | 加權 |
|------|-------------|----------|------|
| 一、功能完整性 | 8 | 8（全部必過） | 30% |
| 二、效能與容量 | 5 | 5（全部必過） | 25% |
| 三、可用性與韌性 | 4 | 4（全部必過） | 20% |
| 四、整合與資料流 | 5 | 5（全部必過） | 15% |
| 五、安全與法遵 | 3 | 3（全部必過） | 10% |
| **合計** | **25** | **25** | **100%** |

**判定規則**：
- 25 項全部 ✅ → POC 通過，雙方進入正式合約簽訂
- 任 1 項 ⚠️ → 條件通過，瑛聲於 14 天內補強並重測，仍通過則進入合約
- 任 1 項 ❌ → POC 失敗，雙方共同檢討是否調整方案或終止 POC

## 三、驗收標準詳列

### 維度一：功能完整性（8 項）

#### F-01｜外撥呼叫成功率

- **要求**：在 POC 期間，外撥呼叫的「成功撥通率」≥ 95%（不含對方未接 / 拒接）
- **量測方法**：CDR 統計，分母為 Outbound Attempts，分子為 Connected
- **量測期間**：完整 4 週 run period
- **失敗處理**：低於 95% 須分析原因（網路 / SIP Trunk / 系統），瑛聲 7 天內提出改善方案

#### F-02｜Inbound 呼叫接聽率

- **要求**：玉山 Pilot 群組之 Inbound 呼叫，「30 秒內接聽率」≥ 80%
- **量測方法**：Speed of Answer (SoA) 平均 ≤ 30 秒
- **量測期間**：完整 4 週 run period
- **失敗處理**：低於 80% 須調整 routing rule 或提升 agent 配置

#### F-03｜致行 Voice Bot 端對端

- **要求**：致行 Bot 完成「開場 + 1 輪客戶問答 + 平滑轉真人」端對端流程，成功率 ≥ 90%
- **量測方法**：抽樣 100 通 Bot 通話，由稽核同事標記「成功」/ 「失敗」
- **失敗定義**：Bot 無回應、轉接時 context 遺失、客戶被切掉

#### F-04｜主管 Whisper 功能

- **要求**：主管可對 Pilot 群組座席進行「私下提示」（客戶聽不到），成功率 100%
- **測試 case**：至少 50 次 Whisper 觸發測試
- **失敗定義**：客戶聽得到、提示音延遲 > 1 秒、權限驗證失敗

#### F-05｜主管 Barge-in / Take-over 功能

- **要求**：Barge-in（三方介入）與 Take-over（接手）功能正常，操作成功率 100%
- **測試 case**：各 20 次

#### F-06｜PCI Pause-Resume

- **要求**：座席手動觸發「暫停錄音」後，錄音檔不含暫停期間音訊；Resume 後錄音續寫
- **測試 case**：至少 30 次 PCI Pause/Resume，每次驗證錄音檔
- **稽核日誌**：每次 Pause/Resume 必須有完整的時間戳 + 操作者 ID

#### F-07｜錄音管理介面

- **要求**：錄音可即時查詢、播放、下載；查詢回應時間 ≤ 3 秒
- **測試 case**：100 次查詢操作（不同條件：日期、座席、客戶號碼、業務類別）

#### F-08｜Hot-desking（彈性座席）

- **要求**：座席可在不同實體位置登入同一個分機，不需重新配置
- **測試 case**：50 次 hot-desking 切換，每次驗證來電可正確路由

### 維度二：效能與容量（5 項）

#### P-01｜並發通話容量

- **要求**：系統可同時處理 50 通並發通話（POC scale），且 CPU / RAM 使用率 ≤ 70%
- **量測方法**：壓力測試工具（如 SIPp）跑 50 並發，持續 30 分鐘
- **量測指標**：CPU avg / max、RAM avg / max、Call Setup Time、Call Quality (MOS)

#### P-02｜並發錄音容量

- **要求**：系統可同時錄製 30 通並發通話，錄音檔完整無遺失
- **量測方法**：壓測 + 抽樣播放
- **失敗定義**：任一錄音檔斷裂、空白、錯位

#### P-03｜致行 Voice Bot 並發

- **要求**：致行 Bot 可同時處理 20 通並發 Bot 通話，回應延遲 ≤ 2 秒
- **量測方法**：壓測 + 端對端 latency 量測

#### P-04｜系統資源使用

- **要求**：在 P-01 並發場景下，系統各層（PBX/VOXEN/Bot）資源使用率均 ≤ 70%
- **量測工具**：Prometheus + Grafana，提供完整 dashboard 供玉山稽核

#### P-05｜錄音 I/O 性能

- **要求**：30 通並發錄音持續寫入 NAS，I/O latency ≤ 50ms（p99）
- **量測方法**：iostat / NAS console 統計

### 維度三：可用性與韌性（4 項）

#### A-01｜HA Failover 實測

- **要求**：3CX 主節點故障，備節點接手 RTO ≤ 90 秒，活通話不掉
- **測試方法**：主節點 systemctl stop，量測接手時間
- **驗證次數**：至少 5 次 failover 演練
- **記錄**：每次 failover 的實際 RTO、活通話狀態（成功 / 抖動 / 掉線）

#### A-02｜計畫性維護不中斷服務

- **要求**：3CX patch 升級可走 rolling upgrade（先升備機，switchover，再升原主），業務不中斷
- **測試方法**：在 POC 期間執行 1 次計畫性 patch 升級

#### A-03｜備份還原驗證

- **要求**：每日備份成功率 100%；任意選 1 個備份點，可在 4 小時內完整還原（不含錄音）
- **測試方法**：DR 演練 1 次

#### A-04｜服務監控與告警

- **要求**：所有元件異常（PBX 服務停止、Trunk 斷線、磁碟滿）皆觸發告警，到達指定通報窗口（Email / Slack / SMS）≤ 5 分鐘
- **測試方法**：人為觸發 10 種異常情境，驗證告警觸達

---

### 維度四：整合與資料流（5 項）

#### I-01｜OPEN/TeleSA 整合

- **要求**：3CX 通話事件可正確觸發 OPEN 的 SCREEN POP，且客戶資料正確
- **測試方法**：100 通通話，每通驗證 OPEN 是否正確跳出 + 資料是否對應
- **失敗定義**：彈窗失敗、資料錯誤、延遲 > 2 秒

#### I-02｜AD/SSO 整合

- **要求**：座席與主管帳號透過玉山 AD 統一驗證，Single Sign-On 一次登入即可使用所有功能
- **測試方法**：50 次登入測試，含正確密碼、錯誤密碼、密碼過期、多因素

#### I-03｜CRM 客戶 360 整合

- **要求**：通話進入時，VOXEN 自動從 CRM 拉客戶 360 資料，呈現於座席畫面
- **量測指標**：拉取延遲 ≤ 1 秒、資料正確率 100%

#### I-04｜SIP Trunk 介接

- **要求**：可順利介接玉山指定的 SIP Trunk（中華電信 / 其他 ITSP），通話品質 MOS ≥ 4.0
- **測試方法**：100 通對外實際撥打，部分由稽核同事評分音質

#### I-05｜資料流向 NAS

- **要求**：所有錄音檔自動歸檔至玉山 NAS，路徑符合玉山命名規則
- **驗證**：抽樣 50 個錄音檔，驗證路徑、檔名、metadata 一致

---

### 維度五：安全與法遵（3 項）

#### S-01｜稽核日誌完整性

- **要求**：所有關鍵操作（登入、錄音播放、PCI Pause、Whisper、Barge-in、Take-over、權限變更）完整記錄於稽核日誌
- **驗證方法**：抽樣 200 個操作，逐一驗證日誌存在 + 內容正確
- **格式要求**：日誌格式須符合金管會 H113-04 規範

#### S-02｜資料加密

- **要求**：錄音檔在傳輸與儲存階段皆加密
  - 傳輸：TLS 1.2+
  - 儲存：AES-256（at rest），key 由玉山保管
- **驗證**：以 Wireshark 確認流量加密 + 直接讀取 NAS 檔案確認加密

#### S-03｜權限分離

- **要求**：管理權限分層 — 系統管理員（瑛聲）、業務管理員（玉山主管）、稽核員（玉山法遵）權限互不交集
- **驗證**：以稽核員帳號嘗試操作管理員功能應被拒絕

## 六、其他條款

### 6.1 POC 期間費用

- POC 期間瑛聲費用全額由瑛聲承擔（含人月、環境、軟體授權）
- POC 通過後若進入正式合約，POC 已建置之環境可作為 Pilot 階段之延續，不重新計費
- POC 失敗，雙方各自承擔自身投入，不互相求償

### 6.2 智慧財產權

- POC 期間瑛聲所交付之文件、程式碼、設定，著作權歸瑛聲，但玉山有「無償內部使用」之永久授權
- 玉山於 POC 期間提供之測試資料，著作權歸玉山，瑛聲僅得用於本 POC

### 6.3 保密義務

- 雙方於 POC 期間取得之對方商業敏感資訊（含但不限於業務流程、技術架構、定價、客戶名單）負保密義務 5 年
- POC 結束後，瑛聲應於 30 天內銷毀所有玉山提供之測試資料，並出具銷毀證明

### 6.4 爭議解決

- POC 過程之技術爭議：雙方共同邀請 3CX 原廠 APAC 技術窗口協助仲裁
- POC 過程之商業爭議：雙方先行協商；如協商不成，依 SOW 主合約之爭議解決條款處理
