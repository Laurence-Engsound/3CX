# 玉山 Pitch 用 Demo 腳本  v1.0

> **適用場合**：玉山銀行 Outbound 客服中心現代化專案 提案會議
> **總時長**：10-15 分鐘（含切換時間）
> **執行人**：瑛聲技術 lead（建議 1 主講 + 1 操作）
> **觀眾**：玉山 IT 團隊
> **撰寫**：瑛聲科技 ・ 2026-04-26

---

## 共通準備（Demo 開始前）

### 環境
- [ ] 3CX V20 Cluster（主備雙機）已在玉山可訪問之 Demo 環境
- [ ] OPEN/TeleSA 連接到 Demo 3CX
- [ ] 致行語音機器人（沿用既有 212 並發授權 5-10 條給 Demo）
- [ ] ADA / SoftPhone 客戶端（座席視角）
- [ ] 主管儀表板（即時監控 + 介入功能）
- [ ] 錄音管理介面（瑛聲既有產品 demo 環境）
- [ ] **2 條測試手機門號** — 模擬「客戶」與「座席手機」
- [ ] 大螢幕 / 投影 + 喇叭（demo 通話聲音要全場聽得到）

### 救命備援（Fail-safe）
- [ ] **每個 Demo 都先錄好影片**（Loom / OBS 螢幕錄影）— 萬一現場連線有問題立刻切影片繼續講
- [ ] 影片放本機 + 雲端 + USB **三份**
- [ ] 簡報筆 / 投影轉接器備品

### 心法
- ✅ **每個 demo 開場一句話 + 結束一句話 takeaway**
- ✅ 失敗時**承認 + 切影片 + 繼續講**，不要假裝沒事
- ❌ 不 demo 後台配置介面（無聊 + 容易出錯）
- ❌ 不 demo 沒準備過的 corner case

## Demo 1：致行（OUTREACH）外撥 + 平滑轉真人

**時長**：5 分鐘
**目的**：證明「**沿用既有 212 bot 授權 + 介接 3CX 完全可行**」，破除「換 PBX 後我們的瑛聲外撥資產要重做」的恐懼

### 開場白
> 「玉山現在有 212 個並發授權的智能外撥語音機器人，我們把他取了一個對外品牌名叫『**致行**』。今天 demo 的是換到 3CX 之後，致行繼續正常工作，而且當客戶想找專員時，可以**平滑無縫**轉接到座席端，且 bot 之前的對話歷程一併移交。」
>
> 「這個 demo 對玉山的意義是 — **您現有的外撥 AI 投資不需要重做**，只是介接層從 Genesys 換成 3CX。」

### 操作步驟
| # | 操作 | 觀眾應該看到 / 聽到 |
|---|---|---|
| 1 | 從 OPEN/TeleSA 觸發外撥任務（測試名單，撥到測試手機 A）| OPEN UI 顯示派號 → 致行接手 |
| 2 | 測試手機 A 響鈴，接起 | 喇叭播放致行 AI 開場白「您好，我是 XX 銀行客服 AI 助理...」|
| 3 | 在電話中口語回覆「我想找專員」 | 致行偵測意圖 → 螢幕顯示「轉接中」|
| 4 | 座席端 ADA 響鈴 + 跳出來電通知 | **重點：來電通知含 bot 已蒐集之資訊**（客戶身分、意圖摘要）|
| 5 | 座席接起 | **重點：座席桌面右側顯示完整 bot 對話歷程逐字稿** |
| 6 | 客戶與座席對話 30 秒 → 結束通話 | 通話結束自動進入 ACW |
| 7 | （備援）顯示通話紀錄已寫入 OPEN/TeleSA + VOXEN 互動歷程 | 雙系統同步寫入 |

### 強調話術（操作中插話）
- 步驟 1：「**這個外撥流程**就是您現在 OPEN 的流程，**沒變**。」
- 步驟 2：「**這就是您現在的致行 bot**。授權數沒變、模型沒變、對話流程沒變。」
- 步驟 4：「**重點在這裡** — 換到 3CX 之後，bot 跟 3CX 的整合一樣穩。Genesys 時代的功能完整保留。」
- 步驟 5：「**這個對話歷程移交**是新增的價值 — 在 Genesys 時代您也許要客戶重述問題，現在不用了。」

### 失敗 fallback
| 失敗模式 | 應對 |
|---|---|
| 致行 bot 不接 | 切影片版 + 「**Demo 環境的 bot 配額用完了，影片是上週實測**」 |
| 平滑轉接失敗 | 「**轉接機制依賴 SIP REFER**，玉山環境的 trunk 配置會在議約後 PoC 階段驗證」+ 跳到下一個操作 |
| 座席端沒收到 bot 對話歷程 | 「**這個整合層在 demo 環境簡化了**，正式版會完整移交，影片可以看到」 |

### 結尾 takeaway
> 「一句話 — **您現在的致行投資 100% 保留**。換到 3CX 不是重做，是**升級介接**。」

## Demo 2：3CX HA Failover 實機演練  ⭐ 最強 demo

**時長**：4 分鐘
**目的**：直擊玉山最大顧慮「系統壞了怎麼辦」 — **眼見為憑勝過 100 頁文件**

### 開場白
> 「玉山被 Genesys 代理商的支援深度傷過。今天我們不講『我們有多可靠』，我們**直接拔網路線**，看看 3CX 怎麼自動接手。」
>
> 「這是現場 demo，**我們對自己有把握**。」

### 操作步驟
| # | 操作 | 觀眾應該看到 / 聽到 |
|---|---|---|
| 1 | 同事用測試手機 B 撥入 demo 3CX 系統 | 主管儀表板顯示「通話中」+ 計時器 |
| 2 | 通話雙方對話幾秒 — 「測試 1 2 3」 | 通話正常 |
| 3 | **拿出網路線 → 拔掉 3CX 主機的網路線**（或 `kubectl drain` 主節點）| 主管儀表板瞬間出現「主機 disconnect」紅色告警 |
| 4 | **靜候 30 秒**（不要急著說話）| 自動 failover 啟動：30 秒內備機接手 |
| 5 | 通話**繼續或重連** — 喇叭測試對方說「還聽得到嗎？」| 喇叭傳出「**還聽得到**」 |
| 6 | 主管儀表板顯示備機已接管，狀態回正常綠色 | 失敗轉移完成 |
| 7 | 重新插回主機網路線 | 主機自動重新加入 cluster（standby 狀態） |

### 強調話術
- 步驟 3：「**重點來了** — 我現在拔的是**主機的網路線**，不是備機。Genesys 同樣場景需要人工介入。」
- 步驟 4：「**靜候**」 — 故意停頓，讓觀眾感受 30 秒（HA 設計目標 ≤ 30 秒）。可以說「**現在備機正在接管**，正常情況 30 秒內。」
- 步驟 5：當對方說「還聽得到嗎？」 → **強調**「通話沒中斷 / 重連」 — **這就是 99.95% SLA 的實際表現**。
- 步驟 7：「主機自動歸隊。**整個過程不需要值班 IT 介入**。」

### 失敗 fallback
| 失敗模式 | 應對 |
|---|---|
| Failover 超過 30 秒 | 「**Demo 環境的監控延遲較高**，正式環境量測為 < 30 秒」+ 切影片 |
| 通話完全中斷無法重連 | 「**這次 demo 環境的 SBC 配置簡化了重連機制**」+ 切影片解釋正式環境的重連設計 |
| 主機拔網路線後無反應 | **絕對不要慌** — 等 60 秒，通常是時間問題。仍無反應則：「Demo 環境網路有狀況，我們切影片」 |

### 結尾 takeaway
> 「一句話 — **這就是 Premium Partner 部署 + 原廠 HA 設計的實際表現**。Genesys 時代 P1 故障要找代理商等回應；3CX 時代很多 P1 場景**根本不會升級到您 IT 的雷達上**。」

## Demo 3：主管即時介入 + 錄音管理介面整合

**時長**：4 分鐘
**目的**：證明「**主管權威 + 瑛聲既有錄音管理介面無縫對接 3CX 後端**」

### 開場白
> 「玉山的主管要看到座席通話、要在關鍵時刻介入、要在事後調聽錄音。我們證明這三件事都行。」
>
> 「**重點在錄音管理介面 — 那是瑛聲既有產品，您現在的調聽流程不會變**，只是後端錄音檔來源從 Genesys 改成 3CX。」

### 操作步驟
| # | 操作 | 觀眾應該看到 / 聽到 |
|---|---|---|
| 1 | 主管儀表板開啟，顯示座席列表 + 即時 KPI | 「目前有 X 通通話進行中，KPI 即時更新」|
| 2 | 同事用手機 C 撥入 demo，與測試座席對話 | 主管儀表板顯示這通通話新增 |
| 3 | 主管按下「Whisper」按鈕，對該通話 | 主管說話 → **只有座席端聽得到**，客戶端聽不到 |
| 4 | 主管 Whisper 提示座席：「告訴客戶我們有優惠方案」| 座席聽到後在通話中執行 |
| 5 | 同事在通話中口頭模擬棘手問題 | 主管按下「Barge-in」 → **三方加入** |
| 6 | 三方對話幾秒 → 主管退出，留座席與客戶 | 通話正常結束，主管已退 |
| 7 | 通話結束 → 切到**錄音管理介面**（瑛聲既有產品） | 錄音管理介面顯示這通通話的紀錄 |
| 8 | 點擊播放 → 完整重播剛才通話 + 標記主管介入段 | 喇叭播出剛才對話 |

### 強調話術
- 步驟 3：「**Whisper 模式** — 主管只跟座席對話，客戶不知道。**這個動作 ≤ 1 秒生效**。」
- 步驟 5：「**Barge-in 是三方** — 棘手客戶或合規場景。整個介入動作 audit log 會留下完整紀錄。」
- 步驟 7：「**重點在這裡** — 這個錄音管理介面就是**您現在用的瑛聲既有產品**。座席操作習慣不變、IT 維運流程不變、合規調聽流程不變。**我們只是把後端錄音檔的來源從 Genesys 換成 3CX**。」
- 步驟 8：「**主管介入段被自動標記** — 質檢時可以快速找到。」

### 失敗 fallback
| 失敗模式 | 應對 |
|---|---|
| Whisper 雙方都聽得到 | 「**Demo 環境的媒體混音模式設定不對**」+ 切影片 |
| Barge-in 失敗 | 跳過此步驟「主管監聽功能視時間決定要不要 demo」 |
| 錄音介面查不到通話 | 「**Demo 環境的 metadata 同步有 5 分鐘延遲**，正式環境 ≤ 5 秒」 |

### 結尾 takeaway
> 「一句話 — **主管功能完整，使用者體驗零學習成本**。錄音介面是您熟悉的瑛聲產品，後端引擎換了但介面層不變。」

## Demo 4（備援）：Last-Agent Routing 場景

**時長**：2-3 分鐘
**前提**：時間夠才 demo（Demo 1+2+3 通常已耗 13 min）
**目的**：證明 VOXEN L5 編排層能完整重建 Genesys URS 的 Last-Agent Routing 邏輯

### 開場白
> 「最後一個 demo — 玉山現在 Genesys 用得最重的功能之一是 Last-Agent Routing：客戶回撥時，系統優先轉給上次接觸的座席。我們證明 VOXEN 編排層能 1:1 對應這個邏輯。」

### 操作步驟
| # | 操作 | 觀眾應該看到 |
|---|---|---|
| 1 | 從測試名單第一筆，模擬「客戶 A」第一次來電 | 系統 skill-based routing 派給座席 X |
| 2 | 通話結束，互動歷程寫入 VOXEN 客戶 360 + 互動歷程資料庫 | 「客戶 A 的最後接觸座席：X」紀錄產生 |
| 3 | 同一測試客戶 A **再次來電**（30 秒後） | 系統先查「客戶 A 上次接觸座席」=「X」|
| 4 | 路由引擎嘗試派給座席 X | 座席 X 端響鈴（**不是隨機派**） |
| 5 | 座席 X 接起 | **重點：通話接通速度 + 座席桌面顯示「客戶上次找您是 30 秒前」** |

### 強調話術
- 步驟 3：「**這個查詢 + 路由動作在 200ms 內完成**。Genesys URS + Stat Server 的延遲類似。」
- 步驟 5：「**座席知道是熟客回撥**，不需要客戶重新解釋脈絡。客戶體驗連貫。」

### 失敗 fallback
| 失敗模式 | 應對 |
|---|---|
| 座席 X 不在線 / 忙碌 | **正好可以 demo 降級邏輯** — 「系統自動降級到 skill-based routing 的次選座席」 |
| 路由錯誤 | 「**客戶 360 的查詢延遲**，正式環境會優化」+ 跳過 |

### 結尾 takeaway
> 「一句話 — **Genesys URS 的核心場景，VOXEN L5 都做得到**。沒有功能損失。」
